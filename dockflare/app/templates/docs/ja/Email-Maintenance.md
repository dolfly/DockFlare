# メンテナンスとトラブルシューティング

DockFlare Email は低メンテナンスで運用できるように設計されていますが、バックアップの処理方法や一般的な問題への対処を理解することは、長期的な信頼性のために重要です。

## バックアップと復元

すべてのメールデータは Docker ボリューム `mail_data` に保存されています。バックアップを実行するには:

1.  **ボリューム全体のバックアップ:** ホストマシン上のボリュームフォルダ全体をバックアップします。これは生の SQLite データベースとすべての添付ファイルを取得できるため、最も安全な方法です。
2.  **UI バックアップ:** **メール** ページで **バックアップと復元** カードを見つけ、**バックアップをダウンロード** をクリックします。これでメールデータの ZIP アーカイブが生成されます。注意: このバックアップにはメールと添付ファイルが平文で含まれているため、安全な場所に保管してください。

復元するには:
1.  `mail_data` ボリュームが `docker-compose.yml` にマウントされていることを確認します。
2.  **メール** ページの **バックアップと復元** カードで、ZIP ファイルを選択して **バックアップを復元** をクリックします。これにより既存のメールデータが完全に上書きされます。

## ログ

配信の問題をデバッグするには、多くの場合 `dockflare-mail-manager` コンテナのログを確認する必要があります。

```bash
docker logs -f dockflare-mail-manager
```

メールページには **配信ログ** カードもあります。**調査** をクリックしてログビューアーを開きます。2 つのタブがあります。
*   **送信ログ:** すべての送信メールの試行履歴。
*   **バウンスログ:** 送信したメールの配信失敗 (NDR) の履歴。

## 耐障害性と自己修復

### R2 バッファリング
サーバーがオフラインになった場合 (停電やインターネット障害など)、Cloudflare Inbound Worker がローカル Webhook に到達できないことを検知します。メールを **R2 の temp_cache** に安全に保管します。
*   ワーカーは 5 分ごとに **Cron ジョブ** を実行します。
*   サーバーが復旧するまで、バッファリングされたメールの配信を自動的に再試行します。

### ファイルシステムのパリティ
Mail Manager には、データベースとファイルシステムが同期していることを確認するスタートアップルーティンが含まれています。添付ファイルが存在するがデータベースレコードがない場合 (「孤立ファイル」)、スペースを節約するために自動的に削除されます。

## 一般的な問題

### ログに「Worker Error」が表示される
API トークンに `Workers Scripts` と `Workers KV Storage` の権限があることを確認してください。DockFlare を最近更新した場合は、メールページで **ワーカーを再デプロイ** をクリックして新しい環境変数を同期する必要があるかもしれません。

### メールが遅延する
Cloudflare Worker ダッシュボードで **Cron** ログを確認してください。ローカルサーバーに高負荷がかかっているか、ネットワークの問題がある場合、ワーカーはメールを R2 にバッファリングし、サーバーが応答したときに配信します。

# パフォーマンスチューニング

ほとんどのユーザーにとって、DockFlare のデフォルト設定はパフォーマンスとリソース使用量のバランスが良好です。ただし、非常に大規模な環境やコンテナの増減が激しい環境では、いくつかの高度なパラメータを調整すると改善する場合があります。

これらの設定は、`docker-compose.yml` ファイル内の環境変数を介して構成されます。

---

## `CLEANUP_INTERVAL_SECONDS`

この変数は、期限切れリソース（猶予期間が過ぎた停止コンテナ由来のルールなど）をクリーンアップするバックグラウンドタスクの実行頻度を制御します。

* **デフォルト:** `60` 秒
* **説明:** 短くするほど古いリソースが Cloudflare 構成から早く消えます。長くするとチェック頻度が下がり、わずかにリソース消費が減ることがあります。
* **いつ調整するか:** 短命コンテナが大量に発生する環境で、後片付けをすぐ行いたい場合は小さくできます（例: `30`）。通常はデフォルトで十分です。

**例:**
```yaml
environment:
  - CLEANUP_INTERVAL_SECONDS=30
```

---

## `MAX_CONCURRENT_DNS_OPS`

この変数は、DockFlare が一度に実行する同時 DNS 操作 (作成、削除) の最大数を設定します。

* **デフォルト:** `3`
* **説明:** サービス数が多い環境向けの、直接的なパフォーマンス調整値です。DockFlare 起動時や大量のコンテナが同時起動したとき、DNS 変更のために Cloudflare API へ送る並列リクエスト数を制限します。
* **いつ調整するか:** 数百規模のサービスを管理していて、初回起動や大規模デプロイで DNS 作成に時間がかかる場合は増やしてみてください（例: `5`、`10`）。上げ過ぎると Cloudflare API のレート制限にかかる可能性があります。

**例:**
```yaml
environment:
  - MAX_CONCURRENT_DNS_OPS=5
```

---

## `RECONCILIATION_BATCH_SIZE`

さまざまなバックグラウンド reconciliation タスクのバッチサイズを制御します。

* **デフォルト:** `3`
* **説明:** 一部のバックグラウンド処理は、システムや Cloudflare API に負荷をかけないようバッチ処理されます。この値はその 1 バッチあたりの件数です。
* **いつ調整するか:** 非常に高度な設定です。ほとんどのケースでは変更しないでください。ルール数が極端に多い（数百〜数千）場合に限り、少し大きくして挙動を確認する価値があります。

**例:**
```yaml
environment:
  - RECONCILIATION_BATCH_SIZE=5
```

---

## `SCAN_ALL_NETWORKS`

DockFlare がコンテナの IP アドレスを見つける方法を変更します。

* **デフォルト:** `false`
* **説明:** デフォルトでは、対象コンテナは DockFlare と同じ Docker ネットワークにいる前提です。`SCAN_ALL_NETWORKS=true` を設定すると、共有ネットワークを見つけるためにコンテナが接続している全ネットワークを検査します。
* **調整する場合:** アプリコンテナが DockFlare と同じネットワークにいない、といった複雑なネットワーク構成のときだけ有効にしてください。検査が増えるため、ネットワーク数が多い環境ではパフォーマンスに影響する可能性があります。

**例:**
```yaml
environment:
  - SCAN_ALL_NETWORKS=true
```
